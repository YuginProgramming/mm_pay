First Product Launch Plan (Simplified and Practical)

Launch goal:
Create a clear, working path for people from the "Corridor" chat:
announcement -> Telegram bot -> profile/access options check -> 500 UAH payment -> terms confirmation -> 1-month access granted.

==================================================
0) Readiness Criteria (Definition of Done)
==================================================

[x] User can open the bot, authorize via email, and see their status.
[x] User can change email anytime via a separate button/command.
[x] User sees available options based on KWIGA data (course/access count).
[x] Bot has a production "Pay 500 UAH" button.
[x] Before payment, user sees usage rules and "I Agree" confirmation button.
[x] After successful payment, access period is stored: start date + end date (30 days).
[x] Active access check logic exists (active/expired).
[x] Key event logging: `wayforpay_webhook_events` (усі виклики webhook), `payment-events.jsonl` (structured), консоль/pm2 для помилок; `debug/list-wayforpay-webhooks.ts` / `debug/list-successful-payments.ts` для перевірки.
[x] Manual fallback: див. §8.1 (orderReference, БД, опційний ручний grant / replay).

2) Data and Infrastructure

2.1 KWIGA data sync
[x] Реалізовано вивантаження KWIGA → PostgreSQL: `database/sync-from-kwiga.ts` (разовий повний прохід: список контактів, для кожного `/contacts/:id/products`, upsert `contacts` / `kwiga_products` / `contact_product_access` з `source = 'kwiga_sync'`).

[ ] Операційно на production: переконатися, що `mm-kwiga-sync` (або еквівалент) запущений, цикли в логах завершуються, помилок KWIGA API немає.
[ ] Перед анонсом: освіжити дані (чек-лист T-24h у §5) і вибірково перевірити вибірку контактів/доступів у БД.
[ ] Вручну перевірити 10–20 реальних email: контакт є в KWIGA → коректний рядок у `contacts` і очікувані рядки доступу в `contact_product_access` (де застосовно).

2.2 Database
[x] Наявність таблиць/полів для:
    [x] email у Telegram-профілі,
    [x] статусу доступу та дат закінчення,
    [x] згоди з правилами,
    [x] подій оплати: `wayforpay_webhook_events`, `pending_wayforpay_orders`, `wayforpay_failure_notices`, grants у `contact_product_access` (`source = 'payment_hook'`),
    [x] дзеркало KWIGA: `contacts` (за `external_id` KWIGA), `kwiga_products`, `contact_product_access` для синху (`source = 'kwiga_sync'`), `app_settings` (у т.ч. `kwiga_sync_interval_minutes`),
    [x] ознаки аудиторії KWIGA у `telegram_users` (`kwiga_audience_rank`, `kwiga_access_row_count`, `kwiga_rank_synced_at`) для профілю бота.
[x] SQL для моніторингу оплат — §2.2.1; для KWIGA — §2.2.2.
[x] Міграції/створення таблиць: `database/migrations.ts` (запуск на сервері після деплою).

2.2.1 Швидкі SQL (моніторинг оплат)
    -- Останні webhook (усі статуси)
    SELECT order_reference, transaction_status, amount, created_at
      FROM wayforpay_webhook_events
     ORDER BY created_at DESC LIMIT 50;

    -- Успішні grant’и з WayForPay
    SELECT id, contact_id, valid_until, amount_paid_uah, wayforpay_order_reference, granted_at
      FROM contact_product_access
     WHERE source = 'payment_hook'
     ORDER BY granted_at DESC LIMIT 30;

    -- Pending інвойси (очікують оплати)
    SELECT order_reference, telegram_user_id, amount_uah, created_at
      FROM pending_wayforpay_orders
     ORDER BY created_at DESC LIMIT 30;

    -- Чи вже відправляли повідомлення про невдачу по orderReference
    SELECT * FROM wayforpay_failure_notices ORDER BY created_at DESC LIMIT 20;

2.2.2 Швидкі SQL (моніторинг KWIGA / контактів)
    -- Скільки контактів з KWIGA в локальній БД
    SELECT COUNT(*) AS contacts_total FROM contacts;

    -- Рядки доступу з синху KWIGA (не оплата WayForPay)
    SELECT COUNT(*) AS kwiga_sync_access_rows FROM contact_product_access WHERE source = 'kwiga_sync';

    -- Найсвіжіша мітка синху по рядках kwiga_sync
    SELECT MAX(last_synced_at) AS last_kwiga_row_sync FROM contact_product_access WHERE source = 'kwiga_sync';

    -- Приклад перевірки одного email (підставити нормалізований email)
    -- SELECT id, external_id, email FROM contacts WHERE email = 'user@example.com';

2.3 Payment flow reliability
[x] Pending-замовлення в PostgreSQL (`pending_wayforpay_orders`), не лише в пам’яті.
[x] Повторні webhook після рестарту: ідемпотентність по `wayforpay_order_reference` + аудит у `wayforpay_webhook_events`.
[x] Webhook idempotency — унікальний індекс / логіка в grant (без подвійного доступу за тим самим orderReference).
[ ] Redis async pipeline — поза поточним scope (див. §1.3; DB-first + sendMessage в payment-процесі).
[ ] Retry / dead-letter для доставки в Telegram — не впроваджено; при збої перевірити логи та §8.1.

4) Pre-Release Test Plan (Mandatory)
==================================================

4.1 Smoke tests (manual, at least 8 scenarios)
[ ] New user -> enters valid email -> sees profile.
[ ] New user -> enters invalid/non-existing email -> sees clear error.
[ ] User changes email -> new email is saved and used.
[ ] User confirms rules -> payment button becomes available.
[ ] 500 UAH payment (success) -> webhook -> 30-day access granted.
[ ] Failed payment -> no access granted, user gets correct message.

4.2 Minimal load check
[ ] 20-50 simultaneous bot entries do not break flow.
[ ] Payment endpoint and bot stay stable and reachable.

8) Plan B (If Something Goes Wrong)
==================================================

[ ] Готовий шаблон повідомлення користувачам при технічних збоях (канал + текст).
[x] Ручне підтвердження оплати та grant — процедура в §8.1.
[ ] Швидкий FAQ для типових питань запуску.

8.1 Ручне підтвердження оплати (fallback)
    1. У WayForPay (або листі клієнта) взяти `orderReference` (UUID платежу).
    2. У БД перевірити: `wayforpay_webhook_events` (чи прийшов Approved), `contact_product_access`
       з цим `wayforpay_order_reference` та `source = 'payment_hook'`.
    3. Зіставити користувача: `pending_wayforpay_orders.order_reference` → `telegram_user_id`, або
       email у `telegram_users` → `contacts`.
    4. Якщо гроші є, а grant-рядка немає — за потреби вручну додати `contact_product_access`
       (або безпечно переотримати/відтворити webhook лише після узгодження процедури; не дублювати доступ).
    5. Повідомити користувача в Telegram вручну або через підтримку (посилання на групи — як у успішному бот-повідомленні).

9) Minimal Implementation Order (Recommended Sequence)
==================================================

Step 1: Profile + available options based on KWIGA data.
Step 2: Stable standalone email change flow.
Step 3: Rules screen + mandatory confirmation.
Step 4: Production "Pay 500 UAH" button + reliable webhook flow.
Step 5: 30-day access assignment and expiration tracking.
Step 6: End-to-end test, then announcement in "Corridor".

==================================================
Launch Go/No-Go
==================================================

If all items in sections 0, 1, and 4 are checked, proceed with first production launch.
If any critical item is missing (payment, access period, rules, profile), delay announcement by 2-5 days and close the gaps first.

==================================================
Production Server Launch Files/Modules
==================================================

For production launch on the server, I need these 3 runtime parts deployed and running:

1) Telegram bot runtime (all Telegram-related running components)
   - `telegram/*`
   - Bot entrypoint and handlers

2) Database / KWIGA sync runtime
   - `database/*` (моделі, міграції, `sync-from-kwiga.ts`)
   - Процес регулярного crawl: `dist/database/kwiga-sync-daemon.js` (PM2 `mm-kwiga-sync`)

3) Payment HTTP server runtime (all payment-related server parts)
   - `payment/*`
   - HTTP server, webhook handling, invoice creation, payment events processing
